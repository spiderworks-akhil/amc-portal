01 Purpose & Background

A single source of truth for every client, website, AMC contract and the infrastructure behind it — that proactively warns us before anything expires or goes down.

We run an agency that maintains client websites under AMC (Annual / Monthly Maintenance Contracts). Today the information lives in spreadsheets and people's heads: which client owns which site, where it's hosted, when the domain or SSL expires, whether the site is up. This CRM centralizes clients → assets → contracts → infrastructure → monitoring → renewals.

Primary goals

  • Single source of truth for clients, websites, contracts and infrastructure.
  • Automatic tracking of domain & SSL expiry — no manual entry.
  • Proactive expiry reminders, early enough to act.
  • Continuous uptime/downtime monitoring with monthly reports.
  • Visibility into AMC renewals and contract value.

02 Core Concepts / Data Model

A flexible hierarchy — nothing is hard-coded to "website". An asset can be any web property type.

03 Functional Requirements

3.1 Client & Asset Management

  • CRUD for clients with multiple contacts each.
  • CRUD for assets; an asset belongs to one client, re-assignable with history.
  • Flexible asset typing — predefined (website, landing page, web app, e-commerce, API, microsite, blog) plus admin-definable custom types.
  • Tags + custom fields on clients and assets.
  • Group / filter by client, type, provider, status, contract, expiry window.
  • Global search + bulk actions + CSV export.

3.2 Contract (AMC) Management

  • Create AMC contracts — monthly or yearly (quarterly/custom too).
  • A contract can cover one or many assets.
  • Track value, currency, scope, start/end, auto-renew.
  • Renewal pipeline — contracts expiring in 30/60/90 days, with reminders.
  • Status auto-transitions (active → expiring → expired) via scheduled jobs; renewal history trail.

3.3 Infrastructure & Provider Management

  • Register service providers (AWS, DigitalOcean, Hetzner, registrars, Cloudflare…).
  • Register servers under a provider; link assets to a server.
  • Track server cost & renewal date (servers expire too).
  • Optional provider API sync (Phase 3).

3.4 Domain Tracking automated

  • Add a domain → WHOIS/RDAP lookup auto-fills registrar, registration, expiry, nameservers.
  • Daily re-check to keep expiry current and detect registrar/NS changes.
  • Flag domain transfers or nameserver changes; expiry countdown.

3.5 SSL Certificate Tracking automated

  • Connect over TLS, read live cert: issuer, validity window, SANs, key type.
  • Daily re-check; compute days-to-expiry.
  • Flag self-signed, host-mismatch, weak or expired certs; distinguish auto-renew (Let's Encrypt) vs manual.

3.6 Uptime / Downtime Monitoring

  • HTTP/HTTPS/keyword/TCP/ping monitors per asset, configurable interval (1–5 min).
  • Record status, response time, status code each check.
  • Auto-open an Incident after N consecutive fails (anti-flap); auto-close on recovery.
  • Track uptime %, MTTR, response-time trend per asset/client.

3.7 Reminders & Notifications

  • Configurable rules per event: domain / SSL / contract / server expiry (+ backup overdue, downtime).
  • Multi-stage reminders (e.g. 30/14/7/1 days before).
  • Email in v1; Slack / WhatsApp / SMS in Phase 2. Recipients: admins, account manager, optionally the client.
  • Escalation if unacknowledged; daily/weekly digest option.

3.8 Maintenance / Ticketing Phase 2

  • Log maintenance work against an asset/contract; lightweight ticketing with SLA & time logging.

3.9 Renewal Tracking no invoicing

  • Track each AMC's value, currency, billing cycle and renewal date.
  • "Expiring soon" pipeline (30/60/90 days) + renewal reminders.
  • Roll-up of total active AMC value for visibility.
Out of scope: invoice generation and payment collection are explicitly not built.

3.11 Authentication & Portal Access minimal

No elaborate role matrix and no multi-tenancy — only the minimal boundary the client portal needs.
  • Two account kinds: Staff (full access) and Client (portal — sees only their own assets, uptime reports and contract/renewal status; read-only).
  • Single agency only — no multi-org / white-label.
  • Email/password login; 2FA optional for staff; basic audit log.

06 Proposed Architecture

Stack

  • Backend: NestJS (TS), REST + OpenAPI/Swagger
  • Frontend: Next.js (App Router), Tailwind + shadcn/ui, TanStack Query
  • DB: PostgreSQL + Prisma
  • Queue: Redis + BullMQ (scheduled jobs)
  • Auth: JWT + refresh, staff/client split, TOTP 2FA
  • Email: Postmark / SES / Resend
  • Storage/PDF: S3-compatible + server-side PDF

Background workers (BullMQ)

  • domain-whois-check — daily
  • ssl-check — daily
  • uptime-monitor — 1–5 min
  • reminder-dispatcher — hourly
  • contract-status — daily
  • monthly-report — 1st of month
  • backup-watch — daily

NestJS module map

auth · users · clients · assets · contracts · providers · servers
domains · ssl · monitoring · incidents · reminders · notifications
reports · dashboard · audit · portal          (Phase 2 adds: tickets)

Integrations by phase

  • Phase 1: WHOIS/RDAP, TLS cert read, HTTP monitoring, SMTP/email.
  • Phase 2: Slack, WhatsApp/SMS (Twilio), Safe Browsing, PageSpeed/Lighthouse.
  • Phase 3: Provider APIs (AWS/DO), status pages, webhooks.
